# Networking Essentials
## OSI模型 7层网络架构
+ 例 web请求
  + tcp三次握手 - 建立网络请求
  + Http get response
  + tcp四次挥手 - 拆除网络请求
+ 三个关键点
  + app开发，可以默认网络自身的可靠
  + 真实的网络请求过程会比较复杂，发生额外的延迟成本
  + 保持可连接的状态其实是消耗成本的

## 传输层
### UDP 快速而不可靠
+ 无连接 + 不可靠 + 无序 + 低延迟
+ 适合速度 > 可靠性的场景，客户可以自行处理丢包或乱序的情况
+ 自定义可靠UDP/QUIC/WebRTC

### TCP 可靠但有额外开销
+ 面向连接 + 可靠 + 有序 + 流式传输
+ 适用数据完整性的场景，即UDP之外的场景

### UDP vs TCP
+ 一般默认适用TCP，需要思考，这道题是否适用UDP
+ UDP适用场景
  + 需要低延迟
  + 允许丢包
  + 海量数据，可以接受偶尔丢包
  + 不需要web支持
+ 现代系统多为TCP+UDP
  + TCP → 控制通道（Control Plane）
  + UDP → 数据通道（Data Plane）

## 应用层
### HTTP 网络基础
+ 通常包括 请求方法，状态码，header，body
+ header一般设计为灵活，而非条件写死
+ 常用API
  + GET 请求资料 幂等
  + POST 发送资料
  + PUT 更新资料
  + PATCH 部分更新
  + DELETE 删除资料 幂等
  > 幂等 重复执行不会改变最终结果

### HTTPS Http加上安全层(TLS/SSL)
> HTTPS 传输过程安全，但是无法保证业务身份
> 需要进行认证和授权
> 认证（Authentication）是否是合法身份
> 授权（Authorization）是否是合法操作

### 常用网络API范式 
+ "我要数据" 一次性
+ REST GraphQL gRPC

#### REST 简单且灵活
+ 原则是讲客户端操作转化为resource资源的操作
+ 默认的API接口设计
+ 后端决定返回什么

#### GraphQL 弹性的资料获取
+ 可以解决欠获取，或者过获取的问题
  + 欠获取 需要多次请求和往返，需要额外的开销和页面加载
  + 过获取 API回应中添加过多的资料，载入变慢
+ 适用场景 需要前端灵活，或多团队高频次查询

#### gRPC 强调性能与低延迟
+ 比HTTP(JSON)更有效的二进制协议
+ 在微服务架构上非常出色
+ 内部API用gRPC 外部API用REST
+ 先解决其他瓶颈再考虑gRPC优化

### 实时通讯
+ "数据持续变化" 长连接，streaming
+ SSE WebSocket WebRTC

### SSE 即时服务器推送
+ HTTP上的改装，允许服务器持续向客户端返回数据
+ SSE的限制，无法长时间进行连接，会自动断开，需要EventSource来重新进行连接
+ 服务器有更新的时候，会主动进行推送
+ 适用场景 即时性较高的场景

### WebSocket 即时双向通讯
+ 基于HTTP
+ 但是需要每一层的基础设施（防火墙，代理，负载平衡器）支持WebSocket
+ 连接流程
  + TCP握手
  + 升级WebSocket
  + C/S双向二进制通讯
  + 通讯保持持续开放直到明确关闭
+ 适用场景 C/S 高频，持久，双向通讯
+ 基础要求较高，不做为推荐方案

### WebRTC 点对点通讯
+ 浏览器之间p2p通讯实现，不需要服务器参与
+ UDP底层实现
+ NAT的ip转化解码
  + stun 介绍双方的地址，并通过打洞的方式穿过NAT
  + turn 中转服务器
+ 连接流程
  + 连接服务器，介绍peer
  + 尝试STUN获取IP
  + 通过服务器分析所需数据
  + 建立点对点连接
  + 需要保留TURN做为fallback
+ 适用场景 音视讯通话或会议
+ 成功率较低，极其不推荐

## 负载平衡 (Load Balancing)
+ 垂直扩展，更大的服务器，业务实战重点
+ 水平扩展，更多的服务器，面试重点
### 客户端负载平衡
+ 非常高效，轮询更新服务器列表，定期同步
+ 例 RedisCluster，节点维护gossip协议，根据客户端请求中的key来分配目标节点
+ 例 DNS，每次返回不同的轮询list，避免其中一个出故障导致整体不可用
+ 适用场景 少量可控客户端或大量客户端但是可以允许缓慢更新

### 专用负载平衡
+ 介于客户端和后端服务器之间的服务器，来分配客户端请求
+ 需要额外的请求次数，但是对服务器更快的更新和颗粒度控制

### L4负载平衡
+ 在传输层根据IP和port做路由决策
+ 特性
  + 持久的TCP连接
  + 检测量最少，所以速度快且高效
  + 无法根据应用层做决策
  + 原始效能优先考虑
+ 适用场景 WebSocket
  
### L7负载平衡
+ 应用层，根据HttpGET理解并智能分配服务器
+ 特性
  + 终止当前连接，向目标后端建立新的连接
  + 根据请求路由
  + 封包检查量大，CPU消耗高
  + 更加灵活和更多的功能
  + 基于Http的浏览
+ 例 将API请求路由到一组服务器，网页请求路由到另外一组，或者根据cookie发到同一台服务器
+ 适用场景 WebSocket之外的所有协议

### 健康检查和容错
+ ⾃动故障切换能⼒，包括TCP检查和L7健康检查

### 负载平衡算法
+ 轮询，随机，最少连接，最快回应，IP哈希

### 实际选择
+ 软件LB，硬件LB（常用），云端LB

## 常见问题
### 区域化和延迟
+ 数据局部性原则 当计算靠近数据的时候，性能最高
+ 内容传递网络CDN 可缓存且需要全球获取信息
+ 区域切片 按照区域切分数据，保证shuju

### 处理故障和失败模式
+ **必须假设网络不可靠**，会出现失败、延迟、异常
+ 超时和回退重试
  + 重试是一个debug的好方法
  + 但是需要添加backoff策略，以及随机jitter
  + **指数退避重试（retry with exponential backoff）**

### 幂等性（Idempotency）
+ 多次请求但结果相同
+ 例如支付系统，多次请求，但是只生效一次
+ 实现方法 幂等键

### 熔断器
+ 级联故障解决方案
+ 正常Closed -> Open -> Half-Open -> test -> Closed/Open
+ 特点
  + 快速失败
  + 减少负载
  + 自我修复
  + 改善用户体验
  + 系统稳定性
+ 适用场景 可靠性、故障模式、灾难恢复等方面
+ 具体有 外部API调用、数据库连接和查找、微服务架构中的服务通信、资源密集型操作、失败或变慢的网络调用
